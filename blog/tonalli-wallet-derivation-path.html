Seed ≠ Wallet: the Derivation Path Mistake That Looks Like “Missing Funds”

In eCash, a seed phrase is only half the story. The other half is the HD derivation path. Mix products with different paths and you don’t get theft — you get a different address.

eCash wallets UX + security xolosArmy: build for clarity

The Core Rule

A wallet is not “seed only.” A wallet is seed + derivation path. Change the path, change the resulting addresses.

“A seed phrase without a derivation path is not a wallet.”

Why people think tokens were stolen

A common failure mode: someone creates a wallet in one product, deposits funds, then imports the same seed into another product and expects the same address. If the derivation paths differ, the address differs — and the funds appear “gone.”

They’re not gone. They’re sitting exactly where they were deposited — on the original derived address.

Tonalli’s position (xolosArmy)

Tonalli chooses a clear, sovereign standard: we keep the original eCash path (m/44'/899'/0'/0/0). That means Tonalli will not silently “match” CashTab addresses — and that’s intentional.

Important: importing a CashTab seed into Tonalli will produce a different address. This is not a bug. It’s deterministic math.

Builder Checklist

How to prevent this class of “missing funds” incidents:

1) Declare the path everywhere
Show it in onboarding + docs + exports
2) Never change paths silently
Path changes create new addresses
3) Add import warnings
“CashTab seed → different address in Tonalli”
4) Support flow (one-liner)
“Funds aren’t gone — you’re on a different derived address.”

xolosArmy Take

Sovereignty requires precision. A wallet that is explicit about derivation is a wallet that protects users — technically and psychologically.
